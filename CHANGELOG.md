# Changelog

All notable changes to this project are documented here.

Format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/); versions
follow [semver](https://semver.org/), with the CLI-specific meaning recorded in
[docs/07](docs/07-distribution-and-roadmap.md#2-versioning--the-template-drift-problem):
**patch** = fixes, **minor** = new commands or template capabilities, **major** =
a convention change that would make `doctor` disagree with projects generated by
an older version.

## [2.2.0] — 2026-07-24

A DEV-ONLY in-app network inspector now ships in the golden template — the web
analog of Chucker's OkHttp interceptor. Every generated project gets it from the
first commit, with nothing to wire.

### Network inspector

Every call through `backendClient` is recorded and shown in a floating overlay:
method, path, status, timing, and the request/response bodies — without opening
the browser's Network tab. Failed calls also raise a toast, so a broken request
announces itself instead of hiding in a panel you have to remember to open.

- `lib/http/net-log.ts` — a bounded ring buffer (100 entries) with a tiny
  pub/sub, consumed via `useSyncExternalStore`.
- `components/dev/network-inspector.tsx` + `dev-tools.tsx` — the overlay,
  dynamically imported by `AppProviders` behind an `isDevelopment` check.
- `backend-client.ts` records both `request` and `requestBlob`, at every exit
  (network / api / parse failure and success). FormData is summarised to its
  field names rather than retained as a blob.

Gated by `isNetLogEnabled` (`NODE_ENV !== "production" && typeof window !==
"undefined"`), a compile-time constant that folds to `false` in production, so
the whole mechanism — and the bodies it would hold in memory — tree-shakes out
of prod builds and never runs on the server.

- `config/env.ts` gains an exported `isDevelopment`, the sanctioned way for app
  code to gate dev tooling without another module touching `process.env`.

Minor, not major: no `doctor` check changed, so projects generated by 2.1.x
remain valid.

## [2.1.0] — 2026-07-23

Fixes from the first round of external tester feedback. All of them are the same
shape: something the architecture *documents* wasn't something the scaffold
*enforced* or *described accurately*.

### The architecture guide now matches the project it ships with

`CLAUDE.md` was written for role-first projects and copied unchanged into
roleless ones, where it described `features/<role>/<domain>/`, `RoleScreens`,
`constants/roles.ts`, `RequireRole` and `features/common/auth` — none of which
exist there — and rendered a half-finished sentence, "roles here:  (plus
`common`)", from an empty role list. An agent or a new developer following it
built the wrong structure.

- The guide is now role-conditional, using the same marker mechanism the rest of
  the template uses. Markdown regions are supported alongside code and JSX ones.
- A roleless project gets a **Single audience** section instead of **Multi-role**,
  stating plainly that the role machinery is absent, and noting that
  `jinn-web role <name>` migrates the project when a second audience is real.
- The project facts (roles, locales) moved to their own line, so neither shape
  has to be grammatical in a sentence written for the other.

### The guardrail pack ships with generated projects

Created projects got a *copied* `eslint.config.mjs` whose header promised
"shared layers never import features" while only the `guardrails` command
actually generated that rule. `lib/` or `components/` importing `@/features/*`
passed lint — which is exactly how layering erodes under a deadline.

- `create` now GENERATES the config from the same renderer `guardrails` uses, so
  the import-boundary zones name the project's real roles.
- `role` and `remove-role` refresh those zones, and the roleless→roles migration
  gets them for the first time. A config that isn't recognisably this pack is
  left alone.
- The shared-layer zone also covers `components/layout/`, `i18n/`,
  `validations/` and `config/`.
- A test asserts the template's own copy is byte-identical to the renderer's
  output, so the two can't drift apart again.

### Template fixes

- **Query errors were reported twice in development.** `attachQueryErrorReporting`
  ran from a `useState` initializer, which Strict Mode double-invokes, attaching
  two subscribers to the browser's singleton client. It's idempotent now.
- **A crash in one screen took out the whole shell.** Added `(app)/error.tsx`, so
  a failing screen is scoped to the content area and the sidebar stays alive.
- **The session cookie name had two defaults** — one in `config/env.ts`, one
  inlined in `proxy.ts`. Both now read `DEFAULT_SESSION_COOKIE_NAME`; the proxy
  still doesn't import `config/env`, deliberately.
- Stale `middleware.ts` references (Next 16 renamed it `proxy.ts`) in
  `guards.tsx`, `(app)/layout.tsx` and `routes.ts`.

### Code quality guidance

`CLAUDE.md` gained a **Code quality** section: definition of done (verify is the
floor, a browser check is the bar), TypeScript rules, React rules (no fetching in
effects, derive don't synchronise, thin pages), naming and file-suffix
conventions, error handling, what to unit-test and what not to, performance, and
the commit workflow. It also documents the backend envelope the app expects and
names the two files to adapt when yours differs.

## [2.0.0] — 2026-07-20

### Domains own a model, and repositories validate responses

Every domain now generates `models/<entity>.model.ts`: a Zod schema plus the
type inferred from it. Repositories parse in **both** directions — they already
parsed what they sent, and now parse what they receive.

The gap this closes: `backendClient.get<Order>(…)` was only ever a claim.
TypeScript erases it, so a backend that renamed a field handed the app
`undefined`, which surfaced as a crash in a date formatter or a currency helper
several layers away — with a stack trace pointing at the formatter, not the
endpoint. Failures now read `GET /orders → items.3.createdAt: expected string,
received null`, at the one place that knows both the endpoint and the field.

- New `models/` folder per domain, holding the wire shapes. `validations/` keeps
  its existing job — form input — because the two genuinely differ: a create
  payload has no `id`, and a wire model has no i18n message keys.
- Two schemas per model. The full entity, and a `…ListItemSchema` derived with
  `.pick()` for table rows. Model the *response*, not the entity: one shared
  shape per entity is what makes a single screen's new field force the backend
  to enrich every list endpoint returning it.
- `types/index.ts` re-exports the inferred types, so existing imports are
  unchanged. Types are never hand-written beside a schema — only `z.infer`.
- `RequestOptions.parse` on `backendClient`, and the same on `serverRequest` so
  Route Handlers aren't an unvalidated back door. `ParseError` joins `ApiError`
  and `NetworkError`; React Query no longer retries it, since a shape that
  failed once fails identically.
- The auth payload and user are schemas now. An unvalidated login response with
  an unknown role produced a user that every `Record<Role, …>` lookup silently
  missed — a parsing bug that presented as a permissions bug.
- Pagination is validated too, replacing a `?? {}` that turned a malformed block
  into an empty object.

Unknown keys are stripped, so a backend **adding** fields still can't break a
client. Only a field you depend on changing shape does.

### Breaking

`doctor` gains a sixth check — repositories that call `backendClient` without a
`parse`. Projects generated by 1.x will fail it until their repositories are
updated (a few lines each, and the report names every one). There's no codemod:
unlike the roleless→roles migration, this needs your real entity shape, which a
tool can't infer.

## [1.1.0] — 2026-07-20

### Roleless projects

`create --no-roles` produces an app with no `Role` type, no `RoleGroup`, no
guards and no role segment in its folders. Features live directly under
`features/<domain>/`, and `domain <name>` needs no `--role`. Many apps have a
single audience, and making them carry role plumbing is pure ceremony.

Twelve template files needed a second shape. Rather than two copies (which
drift) or one brittle subtractive regex per file, the template MARKS the
regions — `jinn-web:role-only` is dropped for roleless projects, and
`jinn-web:roleless` blocks, commented out so the template still compiles and can
be run and swept, are activated. One transform, each region documenting itself.

### Migration, so `--no-roles` isn't a one-way door

`role <name>` on a roleless project converts it: restores the role-shaped files,
moves existing features under `common/` (they were reachable by everyone, so
they're shared by definition), repoints their imports, and adds the new role.

It runs as a transaction — a failure restores the tree — and deliberately does
NOT rewrite service hooks to pass a role group to their query keys. That
signature changes, so the compiler flags every call site, and `role` already
reports compiler errors as a checklist.

### Endpoint grouping decoupled from roles

`domain --grouped` is now its own switch, off by default. Role-scoped **query
keys** stop one role's cache serving another's rows and apply whenever roles
exist; grouped **endpoints** are a fact about the backend's URL shape. Conflating
them made every generated repository carry a group parameter it didn't need.

### Notable fixes

- The first rollback implementation deleted `src/` before restoring the backup,
  and swallowed both errors. When the delete partially failed, the restore had
  nowhere to go and the tree was left with no `src/` at all — a safety mechanism
  causing data loss. It now renames the broken tree aside before moving the
  backup in, and a failed rollback reports where the backup is rather than
  failing silently.
- Restoring `query-keys.ts` from the template destroyed the user's key
  factories. Both it and `nav-config.ts` accumulate generated domains, so both
  get targeted edits to the user's own file.
- The nav item regex matched the `NavItem` interface, which declares `href` too.
- The `api.ts` specifier built `@/features//posts` from an empty role segment.

## [1.0.0] — 2026-07-19

Published as **`jinn-web`**. The architecture it generates is Codeable's
convention set — the tool is a personal project and its name says so.


First complete release. Everything below has been verified by generating a
project and running the full gate on it, not by inspection.

### Commands

- **`create`** — scaffold a project with per-project roles, locales, brand
  colour and API URL. Verifies its own output with `tsc` before handing it over.
- **`domain` / `remove-domain`** — a full CRUD domain (nine files) wired into
  six registries: routes, query keys, nav, the endpoint index and every locale
  catalog. Removal is the exact inverse, asserted by hashing the whole tree.
- **`role` / `remove-role`** — extend `ROLES` and every map keyed off it, then
  run `tsc` and report the remaining errors as a decision checklist. Removal
  refuses while the role still owns domains.
- **`component`** — placed at the right layering tier (domain → role `_shared`
  → cross-role `components/shared`).
- **`dialog`** — confirmation or custom, inside a domain.
- **`add-locale`** — seeds a catalog from the source language with every string
  marked `TODO(<code>)`, and registers it.
- **`rename`** — package and display name.
- **`doctor`** — five static checks over the wiring generators create, with
  `--fix` for the mechanical ones and `--json` for CI.
- **`guardrails`** — install the architecture lint rules into any project,
  including one that predates the CLI. `--level warn` to adopt gradually.

### Template

Next.js 16 · React 19 · TypeScript strict · Tailwind v4 tokens (dark-ready) ·
shadcn/ui · TanStack Query + Table · Zustand · Zod · nuqs URL state · typed i18n.

- **BFF cookie auth** — refresh token in an httpOnly cookie via a thin
  `/api/session` handler, access token in memory. Enables server-side gating in
  `proxy.ts` and keeps a long-lived credential out of reach of XSS.
- **Role-first features** with exhaustive `Record<Role, …>` maps and declarative
  `<RoleScreens>` dispatch.
- ~35 shared components, 11 form fields, a DataTable system on URL state, and a
  helper layer for dates, formatting, mutations and reporting.
- ESLint guardrails, Vitest units, and a Playwright sweep for the defects static
  analysis can't see — layout shear, unreachable truncation, unnamed controls.

### Notable fixes found by verifying output rather than reading code

- The message catalog was `as const`, making every value a literal type, so no
  second locale could satisfy `Messages`. Only visible once a project was
  generated with two languages.
- The brand ramp anchored on the nearest step by lightness, so `#2563EB` landed
  on 700 and `#10B981` on 500 while `--primary` maps to `brand-600` — the app's
  primary colour was never actually the brand colour.
- `TruncatedText` measured itself correctly, set its clipped flag, then
  re-parented the span into the tooltip trigger; that detached the node its
  `ResizeObserver` watched, which fired at 0×0 and reset the flag. Truncated
  values silently never got a tooltip.
- `addNamedImport` rebuilt import lists from names alone, dropping the `type`
  modifier and turning type-only imports into value imports.
- `guardrails` asked ESLint for a formatter removed from core, then read the
  empty output as "no violations" — a false green in a tool whose job is finding
  problems.
- CI ran `npm ci`, which requires the lockfile to describe the install target
  exactly; a lockfile written on macOS/arm64 doesn't carry every Linux optional
  binary, so every run was red from the day the pipeline landed.

### Known limitations

- Publishing to npm is not yet done; install from the repository.
- The template ships no reset-password screen (the backend contract for it
  varies too much to guess).
- `doctor`'s catalog parity check is a belt to TypeScript's braces — it parses
  catalogs textually, so a deliberately exotic catalog shape could confuse it.
